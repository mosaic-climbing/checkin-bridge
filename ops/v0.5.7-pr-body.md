# v0.5.7 — unconditional recheck pass + local-cache needs-match search

## Why

Two separate bugs that v0.5.6 production logs surfaced once the parser fix started landing emails.

### 1. `hydrated > 0` gate at `sync.go:267` suppresses recheck when emails arrive via LIST

v0.5.5's recheck pass promotes pending rows whose UA-Hub user now lands a single-hit match in `cache.customers`. It was gated on `if hydrated > 0` under the assumption that new emails only arrive via the per-user FetchUser hydrate pass.

v0.5.6 broke that assumption: with the `user_email` parser fix, UA-Hub's LIST endpoint started returning emails directly for some users. Production logs confirm it:

```
13:16  listEmails=5   hydrated=0  rechecked=0
16:48  listEmails=52  hydrated=0  rechecked=0  ← +47 emails via LIST, recheck still skipped
```

Ainsley Rae Lightcap is the observed case — her email was populated in `ua_users` by the LIST parser change, she has exactly one matching `cache.customers` row, but her pending record stayed stuck because `hydrated=0` gated off the recheck. The same trap caught every one of the 46 other LIST-delivered emails.

### 2. Needs-Match search hits live Redpoint, fails during 429 storms and misses common name shapes

The Needs-Match detail panel's search box called `redpoint.SearchCustomersByName` — a live GraphQL query using Redpoint's "Last, First" filter. Three failure modes staff ran into:

- No email path at all. If staff knew a member's email, they had to guess at the name and try to outthink Redpoint's filter.
- Single-token queries ("dana") sent as `Last: , First: dana`, which Redpoint rejects for names that are actually surnames (and vice versa).
- Redpoint 429s or latency spikes (the Apr 20 storm is ongoing as of writing — tracked separately as task #109) blind the whole panel.

The `customers_fts` FTS5 index added in migration 3 already covers name + email + external_id + barcode with prefix-AND semantics. Point the search box at it.

## What

Two commits on this branch:

### 1. `v0.5.7: drop hydrated>0 gate on recheck pass`
- `internal/unifimirror/sync.go` — removes the `if hydrated > 0` wrapper; `recheckPending` runs on every refresh. Postmortem comment on the call site so the gate doesn't come back as a "cheap optimization" in a future PR.
- `internal/unifimirror/sync_test.go` — adds `TestRefresh_RecheckRunsWhenListDeliversEmail` (asserts `Hydrated=0, Rechecked=1` when a LIST-delivered email lands a single-hit match) and `TestRefresh_RecheckIsNoOpWithNoMatches` (pins that the always-on pass is a no-op when there's nothing to promote — protects the steady-state cost).

### 2. `v0.5.7: needs-match search uses local cache FTS (name + email)`
- `internal/api/server.go` — `renderNeedsMatchDetail` switches from `s.redpoint.SearchCustomersByName` to `s.store.SearchCustomersFTS`. Removes the now-unused `splitName` helper. Adapts `store.Customer` → `ui.NeedsMatchCandidate` directly, using the mirror-populated `BadgeName`/`BadgeStatus` columns instead of the Redpoint-client `badgeNameFor`/`badgeStatusFor` helpers.
- `internal/ui/fragments.go` — placeholder text now reads "Search Redpoint by name or email…".
- `internal/api/needs_match_test.go` — three new tests: search by email hits the local cache and excludes non-matches; single-token name with a hyphenated last name (the old path's worst failure case) resolves via FTS prefix-AND; multi-word queries AND correctly across tokens.

## Expected post-deploy outcome

1. After the next `POST /ua-hub/sync`, Ainsley Rae Lightcap (and up to 46 other LIST-delivered-email users with single-hit Redpoint matches) auto-promote out of `ua_user_mappings_pending` into `ua_user_mappings` with `matched_by='auto:email:recheck'`. JSON response shows `rechecked >= 1` where v0.5.6 always returned `rechecked=0`.
2. The Needs-Match detail-panel search box accepts emails and resolves single-token / hyphenated / multi-word name queries that the live-Redpoint path was missing. Search continues to work even while Redpoint is in the 429 storm (task #109).
3. `hydrated` stays at 0 for LEF — legitimately, because UA-Hub genuinely does not have `user_email` populated for the remaining 1566 users. That's a data shape at the source, not a bug in the bridge.
