# v0.5.9: member management overhaul — remove Add form, add detail panel with Unbind / Reactivate / Reassign / Remove

Single-ticket scope focused on one pain point surfaced after v0.5.8
went to LEF: the Members page had no recovery path when a binding went
sideways. If the auto-matcher pointed a UA-Hub user at the wrong
Redpoint customer (household email collision, hyphenated-name
disambiguator miss, operator skipped the wrong row in Needs Match),
staff had no way to fix it short of SQLite surgery. v0.5.8's `Remove`
button dropped the whole cache row, which was overkill for
"re-bind to the correct customer" and under-kill for "move this card
to a different member". Meanwhile the bridge-side Add Member form had
accumulated its own pile of edge cases (O'Brien rendered a broken
button, #118) and was the wrong abstraction in the first place — at
LEF, UA-Hub is the system of record for identity and the bridge's job
is to stay in sync with it. Creating users on the bridge side meant
two write paths diverging at 3am.

v0.5.9 deletes the Add form outright and replaces the per-row `Remove`
button with a detail panel that carries four distinct recovery
actions. Each action maps to exactly one class of misbinding, so staff
picks the button that matches what actually went wrong instead of
nuking the whole row and re-running a sync.

## The four recovery actions

1. **Unbind (re-queue)** — keeps the UA-Hub user and the bridge cache
   row, clears the `ua_user_mappings` row, re-queues the user in
   Needs Match with `PendingReasonNoMatch`. Use when the auto-matcher
   bound the wrong Redpoint customer and you want to pick a different
   candidate from Needs Match. Writes a single
   `match_audit` row: `field=mapping`, `before=<old customer>`,
   `after=""`, `source=staff:unbind`.

2. **Reactivate UA-Hub user** — flips UA-Hub status from
   `DEACTIVATED` back to `ACTIVE`. Use when staff hit `Skip` in Needs
   Match and later realised it was the wrong call. Calls UA-Hub
   `UpdateUserStatus` + writes `match_audit` row with
   `field=user_status`, `before=DEACTIVATED`, `after=ACTIVE`,
   `source=staff:reactivate`. Button only enables when
   `UAUser.Status == "DEACTIVATED"` — otherwise it's a tooltip-only
   disabled chip so staff knows exactly why ("User is not
   deactivated"). Tolerant of a missing UA-Hub mirror row:
   `before_val` records as "unknown" and the reactivate still fires
   (UA-Hub `UpdateUserStatus` is idempotent for already-ACTIVE
   users).

3. **Reassign NFC card** — the big new one (#10). Moves an NFC token
   from one UA-Hub user to another using UA-Hub §3.7 `AssignNFCCard`
   with `force_add=true`. UA-Hub handles the three-step atomic
   (remove from current owner, add to target, return the updated
   target row). Two `match_audit` rows per reassign — one keyed on
   the old owner's UAUserID (`field=nfc_card`, `before=<token>`,
   `after=""`) and one keyed on the new (`before=""`,
   `after=<token>`) — both tagged `source=staff:reassign`. Either
   user's forensic view surfaces the hand-off. The bridge also
   rewrites the `members` row's `customer_id` + identity fields to
   match the new UA-Hub user's mapping (if one exists), so the
   Members table shows the correct name for the card straight away.

   UX: three-handler sequence via the existing
   `/ui/frag/member/{nfcUid}/reassign(/search|/confirm)` routes.
   Panel-level swap into `#member-detail`: picker → search → confirm
   → success alert, with a `Cancel` button on every screen that
   hx-GETs the detail fragment back. Each candidate row's `hx-confirm`
   spells out the two-sided effect ("Move card X from A to B?"),
   and rows whose UA-Hub user already has a card carry a ⚠ hint so
   staff sees the three-way swap coming.

4. **Remove from cache** — the pre-existing escape hatch (DELETE
   `/members/{nfcUid}`), re-shaped to match the new panel model. The
   UA-Hub user survives; the bridge member row goes away and the
   next ingest either re-binds the user or drops them into Needs
   Match. No audit row (the mapping row goes with the DELETE and
   there's nothing to pin an audit entry to).

## What's gone

The `/ui/members/new` route, its handler, its form template, and the
config knobs that gated it. UA-Hub is the source of truth for
identity; the bridge stays a sync consumer. Staff still creates
users in UA-Hub's native admin; the next bridge sync auto-binds them
to Redpoint (email+name match) or drops them in Needs Match for
manual pairing.

Members page also got a banner at the top explaining the new flow
("Adding a new member? Create the user in UniFi Access first …") so
nobody wonders where the Add button went. The banner links to Needs
Match so staff has a direct jump when the auto-bind doesn't fire.

## Sort order change (#6)

`AllMembersPaged` now sorts by `ua_user_mappings.matched_at DESC`
with NULLs (orphans) at the bottom, alphabetical tiebreak on
`(last_name, first_name)`. Rationale: after a bulk ingest the
newly-bound members are the ones staff is most likely to open the
detail panel on — misassignments surface at the top of the list
rather than needing a scroll through the alphabetical back-catalogue.
Orphans are historical debris; the recovery action is `Remove` and
position doesn't matter. SQLite doesn't support `NULLS LAST` so the
query emulates it with `ORDER BY map.matched_at IS NULL,
map.matched_at DESC, ...`.

## What ships

### Member detail panel + three new handlers

- `internal/api/members_detail.go` (new, ~290 lines):
  `handleFragMemberDetail` / `handleFragMemberUnbind` /
  `handleFragMemberReactivate`. All three emit
  `HX-Trigger: member-updated` on success so the member table
  auto-refreshes. Unbind writes its audit row *before* deleting the
  mapping so the forensic trail records even if the delete fails.
- `internal/api/members_reassign.go` (new, ~230 lines): the three
  reassign handlers share a `resolveReassignContext` helper that
  centralises the member + mapping + UA-mirror lookup chain and the
  "which error fragment to render" logic. Saves each handler from
  re-implementing the same 40-line preamble.
- `internal/ui/fragments.go`: new `MemberDetailFragment` (+ view-
  model structs) and `MemberReassignFragment` (+ view-model structs).
  Button-enabling rules documented on the view-model types so
  `handleFragMemberDetail` and the fragment stay in sync. Disabled
  buttons still render with a tooltip explaining why, so staff
  doesn't wonder "where did the button go" on a sync-lag row.
- `internal/ui/templates/pages/members.html`: new
  `#member-detail` sticky sink above the member table; `View details`
  buttons in the search-results table and per-row buttons in the
  member table both hx-get into that sink. Banner explaining the new
  provisioning flow and linking to Needs Match.

### Sort + store additions

- `internal/store/members.go`: `AllMembersPaged` query rewrite
  (see above). Includes a comment block pinning the sort contract
  so nobody simplifies the join back.
- `internal/store/ua_users.go`: new `SearchUAUsers(ctx, q, limit)`
  — case-insensitive LIKE across `first_name`/`last_name`/`name`/
  `email`, tokenised on whitespace and AND-ed. Used by the reassign
  picker. Deliberately a LIKE walk rather than FTS5 — the ua_users
  mirror tops out in the low thousands at LEF's scale and the
  reassign picker is cold-path; an FTS index here would be premature
  optimisation + an extra migration.
- `internal/statusync/matcher.go`: new `MatchSourceStaffUnbind` /
  `MatchSourceStaffReactivate` / `MatchSourceStaffReassign`
  constants so audit queries can count misassignment recoveries
  separately from first-time matches.

### Removed

- `internal/api/members_new.go` (-533 lines), its test file
  (-579 lines), and `internal/ui/templates/pages/members-new.html`
  (-47 lines).
- `internal/config/config.go`: `AllowNewMembers`,
  `DefaultAccessPolicyIDs`, `RequireMinimumUAHubVersion` and their
  env overrides / validation.
- `internal/api/server.go`: `/ui/members/new`, `/members` POST, and
  the associated CSRF-fetched bootstrap routes.

## Ops follow-up at LEF (after deploy)

- **Smoke-test the detail panel.** Open a known-good member on the
  Members page, click `Details`, and confirm all four buttons render
  (Unbind + Remove enabled, Reactivate disabled with its "User is
  not deactivated" tooltip on an ACTIVE user, Reassign enabled).
  Open an orphan (if any exist) and confirm Unbind + Reactivate are
  disabled with the "No UA-Hub mapping" tooltip.
- **Dry-run a Reassign.** Pick a test card + a test UA-Hub user not
  currently associated with any Redpoint customer. Walk the flow:
  picker → search → confirm → success alert. Tap the card on a
  reader and confirm the check-in logs show the new owner.
  Afterwards, hit `/members` JSON and confirm the member row reflects
  the new `customer_id`. Open Details on the new owner and confirm
  the audit trail shows the two `nfc_card` rows with
  `source=staff:reassign`.
- **Watch for "mirror row missing" reactivates.** The v0.5.9
  fallback records `before_val="unknown"` when a reactivate fires
  on a member whose UA-Hub mirror hasn't synced yet. If this turns
  up more than once or twice per week in `match_audit`, the nightly
  unifimirror sync is lagging and we should look at its schedule /
  health.
- **No manual DB steps needed.** All v0.5.9 changes are in-process;
  no SQLite surgery required. The removed config fields
  (`AllowNewMembers`, `DefaultAccessPolicyIDs`,
  `RequireMinimumUAHubVersion`) will be silently ignored if still
  present in `config.json` — no boot failure.
