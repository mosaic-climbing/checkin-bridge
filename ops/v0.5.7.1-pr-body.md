# v0.5.7.1: sync-page lifecycle + mid-flight progress pill

Patch on top of v0.5.7 that closes three rough edges staff hit on the
`/ui/sync` page at LEF:

1. **`finishSyncJob` no longer inherits the request context.**
   `r.Context()` was passed straight through to `CompleteJob`/`FailJob`,
   but a UA-Hub mirror refresh routinely takes 4–5 minutes — well
   beyond the point HTMX/the browser abandons the long POST. Once the
   request context was `Done()`, the SQLite UPDATE silently no-op'd
   and the `jobs` row stayed pinned at `running` forever. Observed at
   LEF on v0.5.7: three stale rows accumulated across two deploys
   despite log evidence the refresh completed cleanly each time. Fix
   detaches the parent ctx with `context.WithoutCancel` and bounds
   the write with a 5s deadline.

2. **Mid-flight progress in the "Last run" pill.** Pre-patch the pill
   went from `Never run` straight to `⟳ Running · started just now`
   and stayed there for ~5min. Now the Syncer emits phase strings
   (`listing UA-Hub users`, `upserting N list rows`, `hydrating
   450/1500`, `reconciling pending mappings`) into a `ProgressFunc`
   that the api layer wires to `Store.UpdateJobProgress` for the
   in-flight job id. The pill polls every 15s (existing hx-trigger)
   and renders the latest phase inline so staff see actual progress
   instead of a static badge.

3. **One-click escape hatch for stuck rows.** When a `running` job is
   older than `unstickAgeThreshold` (10min), the pill grows a
   `Clear stuck` link that POSTs to `/ui/sync/unstick/{type}`. The
   handler marks the latest `running` row failed with a clear
   reason (`manually cleared via /ui/sync (running row stuck)`) and
   re-renders the pill. No more `ssh bridge → sudo sqlite3` to
   unwedge a card. Defensive against the residual cases that even
   the ctx-cancel fix can't catch (mid-refresh daemon crash, SQLite
   lock contention).

Plus the v0.5.7.0-discovered concurrent-click bug: the staff page now
sets `hx-disabled-elt="this"` on every Run/Dry-Run/Recount button so
two clicks in quick succession can't trigger two parallel refreshes
that race for `FindResolvablePending` (the second one was canceling
the first's recheck and emitting `mirrorTotal=0` in the failed
response, which scared staff even though the original refresh was
still landing rows correctly).

## What ships

### Core fix: detached ctx for terminal-state writes
- `internal/api/sync_ux.go`: `finishSyncJob` now calls
  `context.WithTimeout(context.WithoutCancel(ctx), 5*time.Second)`
  before the `FailJob`/`CompleteJob` call. Comment block explains
  the LEF observation that drove the change.

### Phase/progress reporter
- `internal/unifimirror/sync.go`: adds `ProgressFunc`, `WithProgress`
  context helper, `reportProgress` package-internal accessor.
  `RefreshWithStats` and `upsertAndHydrate` emit phase strings at
  the four meaningful boundaries. Hydrate cadence throttled to
  every 25 rows (`hydrateProgressEvery`) so the SQLite progress
  write rate stays sane on large refreshes.
- `internal/api/server.go`: `uaHubMirrorRefresh` callback signature
  grows a `progress func(phase string)` parameter; setter and
  handler updated. Comment captures the lifetime story.
- `internal/api/sync_ux.go`: `makeJobProgressFn(jobID)` returns a
  closure that detaches its own context (same pattern as
  `finishSyncJob`) and writes to `jobs.progress`. The handler
  builds one per request and forwards into the refresher.
- `cmd/bridge/main.go`: closure now does
  `ctx = unifimirror.WithProgress(ctx, progress)` before invoking
  `RefreshWithStats`.

### Unstick endpoint
- `internal/api/sync_ux.go`: `handleSyncUnstick` — POST handler that
  finds the latest `running` row of a given type, fails it with a
  clear reason, audit-logs the operator action, and re-renders the
  pill via the same `SyncLastRunPillFull` path as the auto-refresh.
  Idempotent: no-running-row → re-renders the latest pill state.
- `internal/api/server.go`: registers
  `POST /ui/sync/unstick/{type}` next to the `/ui/frag/*` cluster.

### Pill rendering
- `internal/ui/fragments.go`:
  - `SyncLastRunPillFull(jobType, status, createdAt, errMsg, progress)`
    — new full-fat renderer. Old `SyncLastRunPill` is now a four-arg
    shim that calls through with empty progress, so legacy call
    sites keep compiling.
  - Running-case branch: shows `⟳ Running · {phase} · started Xm
    ago` when fresh, drops the phase and appends a `Clear stuck`
    HTMX link when older than `unstickAgeThreshold` (10min).
  - `FormatRelative`: now accepts BOTH RFC3339 (Go-side writes)
    and SQLite's `YYYY-MM-DD HH:MM:SS` (the `DEFAULT
    CURRENT_TIMESTAMP` form on `created_at`). Pre-patch this
    silently broke every running pill — `created_at` always
    failed to parse and the pill rendered "never".
  - `parseStoreTimestamp`, `isStuckRunning`, `trimPhase` — small
    helpers, all under the same fragment-block doc comment.

### Concurrent-click guard
- `internal/ui/templates/pages/sync.html`: every Run/Dry-Run/Recount
  button gets `hx-disabled-elt="this"`, so the click is locked
  until the response comes back. Stops two manual clicks from
  racing each other (and stops the page-refresh-scheduler-fired
  refresh-vs-manual-click race that LEF saw at 18:43 UTC).

## Tests

### `internal/unifimirror/progress_test.go` (new)
- `TestRefresh_EmitsPhaseProgress` — pins the always-fires phase
  sequence (listing → upserting → reconciling).
- `TestRefresh_EmitsHydrateProgress` — drops `hydrateProgressEvery`
  to 1 and asserts the slow-path hydrate phase fires.
- `TestRefresh_NoProgressFuncIsNoop` — confirms `Refresh` works
  unchanged when no `WithProgress` is in flight (the ticker path).
- `TestWithProgress_NilFuncReturnsCtx` — pins the documented
  no-op behavior so callers can wrap unconditionally.

### `internal/api/sync_ux_lifecycle_test.go` (new)
- `TestFinishSyncJob_DetachesCancelledContext` — regression test
  for the bug. Pre-fix: row stays `running` after finishSyncJob
  on a cancelled ctx. Post-fix: row transitions to `completed`.
- `TestFinishSyncJob_CancelledContextFailureBranch` — same for
  the `FailJob` path.
- `TestHandleSyncUnstick_RunningRowFailsAndRendersPill` — happy
  path; running row → failed + Failed pill in body.
- `TestHandleSyncUnstick_NoRunningRowIsIdempotent` — covers the
  race where the daemon transitioned the row first; must not
  clobber a finished row.
- `TestHandleSyncUnstick_UnknownTypeIsHarmless` — defensive
  renderer never 5xx's on an arbitrary type segment.
- `TestMakeJobProgressFn_WritesJobsProgress` — pins the seam the
  cmd/bridge closure relies on; phase string lands in
  `jobs.progress`.

### `internal/ui/fragments_sync_pill_test.go` (new)
- `TestFormatRelative_AcceptsRFC3339` / `_AcceptsSQLiteTimestamp`
  / `_EmptyIsNever` / `_UnparseableIsNever` — pin the dual-format
  parser and the "never" fallback.
- `TestSyncLastRunPillFull_RunningWithProgress` — fresh running
  pill shows phase, no unstick link, JSON quotes stripped from
  the rendered phase.
- `TestSyncLastRunPillFull_RunningStuckShowsUnstickLink` — stale
  running pill drops phase, appends `Clear stuck` with the
  correct hx-post target.
- `TestSyncLastRunPillFull_RunningWithoutProgress` — fresh pill

- `TestSyncLastRunPillFull_CompletedUnchanged` — non-running

## Ops follow-up at LEF (do this manually before redeploying)

The three stale `running` rows from v0.5.7 still need clearing
since they were written before the patch landed. Pill should clear
itself on next page load after deploy thanks to the unstick link,
but if you want to be tidy:

```bash
ssh bridge 'sudo sqlite3 /usr/local/mosaic-bridge/data/audit.db \
  "UPDATE jobs SET status=\"failed\", \
                   error=\"manual unstick: ctx-cancel bug (fixed in v0.5.7.1)\", \
                   updated_at=datetime(\"now\") \
   WHERE status=\"running\" AND type=\"ua_hub_sync\";"'
```

Or just click `Clear stuck` on each card after the deploy goes
green — same effect, audited via the new `sync_unstick` audit
log entry.
