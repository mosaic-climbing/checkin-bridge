# v0.5.6 — Hydrate UA-Hub emails from `user_email` + surface mirror stats

## Why

v0.5.5 shipped per-user email hydration + post-refresh pending-row recheck inside `unifimirror.RefreshWithStats`, but after deploy ua_users still showed **5 emails out of 1618 users**. The sync was running — it just wasn't landing emails. Diagnosis turned up two separate bugs:

1. **Root cause — wrong JSON field.** UA-Hub's `/users/{id}` response has two email-ish fields:
   ```
   "email":      ""                              // verification-workflow slot
   "user_email": "ainsleyraelightcap@gmail.com"  // operator-entered address
   ```
   `parseUniFiUser` read `email`, which is empty for 1613/1618 users at LEF. The hydration loop was dutifully fetching each user and storing a blank email. The 5 users who *did* have `email` populated were the only ones that ever worked.

2. **Visibility gap.** `api.UAHubRefreshStats` didn't carry `Hydrated` / `Rechecked`, so the sync response couldn't tell us whether hydration had run-and-found-nothing vs. not-run-at-all. That's how this bug hid through v0.5.5 — and why we almost shipped a v0.5.5.1 tag-fix thinking the earlier build had shipped wrong code. Turned out the code was fine; the *parser* was fine; the *field name* was wrong.

## What

Two commits on the branch:

### 1. `v0.5.6: surface UA-Hub mirror Hydrated/Rechecked in sync response`
- `internal/api/server.go` — `UAHubRefreshStats` gains `Hydrated int` / `Rechecked int`; handler maps them into both the `finishSyncJob` JSON payload and the HTMX `SyncStat` rows.
- `cmd/bridge/main.go` — refresher callback forwards `stats.Hydrated` / `stats.Rechecked` from `unifimirror.Stats`.

### 2. `v0.5.6: fix parseUniFiUser to read user_email (root cause of 5/1618 hydration)`
- `internal/unifi/client.go` — `parseUniFiUser` now prefers `user_email`, falls back to `email` so we don't regress the 5 users with the legacy shape. Postmortem comment in the code so nobody re-introduces this.
- `internal/unifi/client_test.go` — table test for the four field combinations (user_email only / email only / both / neither / missing), plus a regression test against the exact UA-Hub single-user shape observed in production.

Fix is one line of logic; the docs + tests carry the weight so this shape quirk stays documented.

## Expected post-deploy outcome

Run `POST /ua-hub/sync` once after deploy. The JSON response should include `hydrated` and `rechecked` counts, and `ua_users` should flip from 5 → ~1618 populated emails in a single sync cycle.
