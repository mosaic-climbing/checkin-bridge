# v0.5.6 — Surface UA-Hub mirror Hydrated/Rechecked in sync response

## Why

v0.5.5 shipped per-user email hydration + post-refresh pending-row recheck inside `unifimirror.RefreshWithStats`, but the counts never made it out of the sync handler:

- `api.UAHubRefreshStats` (the refresher-callback payload) only carries `Observed / Upserted / MirrorTotal / Duration`.
- `cmd/bridge` maps `unifimirror.Stats` → `api.UAHubRefreshStats` one-liner, dropping the new fields.
- `handleUAHubSync`'s `finishSyncJob` payload, JSON response, and the `[]ui.SyncStat` rows all omit the new fields.

Net effect: `POST /ua-hub/sync` returns `{observed, upserted, mirrorTotal, success}` with no visibility into whether hydration or recheck actually ran. A production sync at LEF (1618 users / 5 emails in list payload) came back fast and with no hydrated-count, so we can't tell whether hydration ran-and-found-nothing or didn't run at all. The database still shows 5/1618 emails after the sync, which points at the latter.

## What

Surgical surfacing change — no logic changes.

- `internal/api/server.go` — `UAHubRefreshStats` gains `Hydrated int` and `Rechecked int`; handler maps them into both the `finishSyncJob` JSON payload and the HTMX `SyncStat` rows.
- `cmd/bridge/main.go` — refresher callback forwards `stats.Hydrated` and `stats.Rechecked` from `unifimirror.Stats`.

Post-merge, `POST /ua-hub/sync` JSON will include `hydrated` and `rechecked`, and the Sync page's UA-Hub tile will show "Hydrated N" / "Rechecked N" rows. That lets us tell on the next manual run whether:

1. Hydration doesn't run at all (stays at 0) → wiring / deploy issue.
2. Hydration runs but FetchUser returns empty emails too (large 0 with long duration) → UA-Hub API quirk, separate follow-up.
3. Hydration works as designed (N ≈ listBlankEmails, Rechecked ≥ 0) → v0.5.5 is correct; action is just to re-run.
