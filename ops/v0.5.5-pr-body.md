## Summary

**Needs Match: backlog-clearing mirror hydration + Open-button layout fix + UA-Hub probe.**

v0.5.4 fixed the *symptom* of Needs Match being useless (345 of 345 rows blank, Open button hidden 24k pixels below the fold). v0.5.5 fixes the *cause* — and turns the accumulated backlog into an empty page without staff clicking 339 times.

### Fix 1 — UA-Hub mirror hydrates blank emails per-user

Diagnosis on live audit.db at LEF showed:

```
ua_users total      : 1618
with name           : 1618
with email          :    5   ← the shape-level bug
```

The paginated `ListAllUsersWithStatus` endpoint returns first/last/name for every row but omits `email` for 99.7% of users. Every downstream email match (ingest matcher, denied-tap recheck, nightly walker) was starving on a mirror that could never answer "who has this email."

v0.5.5 extends `internal/unifimirror.unifiClient` with `FetchUser(ctx, id)` (per-user `GET /users/{id}`, which UA-Hub answers in full) and calls it for any row that comes back from the list endpoint with a blank email. Pacing between calls (75ms; `hydrateInterval` package var so tests can zero it out) — finishes ~1600 users in ~2 minutes, well within a nightly sync window and polite to UA-Hub.

Optimization: if the list endpoint already returned every email (deployment where UA-Hub is richer than LEF's), zero FetchUser calls fire. Tested.

### Fix 2 — Post-refresh pending-row recheck pass

Hydrating the mirror doesn't retroactively visit existing pending rows. Without a recheck pass, the 339-row backlog at LEF would have stayed full; only *new* pending decisions would have benefited from the richer mirror.

New `Syncer.recheckPending` runs after every refresh that actually hydrated something. It walks `ua_user_mappings_pending JOIN ua_users JOIN cache.customers` for rows where:

- `ua_users.email` is now non-blank (i.e. freshly hydrated)
- exactly one customer in cache shares that email (`COUNT(*) = 1`)
- no existing mapping already claims this UA-Hub user or Redpoint customer

Single-hit rows land in `ua_user_mappings` with `matched_by='auto:email:recheck'` (new label; same confidence as `auto:email`, distinct so audits can count how much work the recheck is doing). Pending row is deleted. Audit row appended.

Ambiguous cases (household collision: parent + child sharing an address) are **left alone** — same rule as the statusync matcher. Staff continues to adjudicate those through the Needs Match page.

### Fix 3 — Open button actually lands in view

v0.5.4's `show:top` modifier improved things but Chris reported "it puts the interface at the bottom" — root cause was that the detail-panel sink was below the 345-row table, so `show:top` was scrolling to the top of a 4000px-tall detail panel that was still below the viewport start.

Fix moves `#needs-match-detail` **above** the list, with `position: sticky; top: 0` so scrolling down the list doesn't lose the just-opened detail. The Open button now uses `show:window:top` so clicks anywhere in the list return the operator to the panel.

### Fix 4 — `GET /admin/ua-hub/fetch/{id}` debug endpoint

Control-mux endpoint that returns the bridge's parsed view of any UA-Hub user. Previously the only way to answer "does UA-Hub actually have this email?" was to SSH in and curl UA-Hub with the raw API token — fragile, slow, and a bad fit for on-call. Exposed via control port with the same bearer auth as `/admin/mirror/*`.
